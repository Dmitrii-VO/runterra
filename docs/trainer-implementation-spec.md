# Спецификация реализации функционала "Тренер" (Trainer 2.0)

## 1. Контекст и Цели
Текущая реализация роли `TRAINER` в клубах является номинальной. Цель данной спецификации — превратить тренера в функциональную единицу системы, предоставив инструменты для ведения профиля, создания библиотеки тренировок и управления прогрессом учеников.

Ссылка на ADR: `docs/adr/0005-trainers-role.md`

---

## 2. Архитектура и Сущности

### 2.1. Профиль Тренера (`TrainerProfile`)
Глобальный профиль пользователя как специалиста. Существует в единственном экземпляре для `User`, независимо от количества клубов, в которых он состоит.

**Таблица:** `trainer_profiles`

| Поле | Тип | Описание |
|---|---|---|
| `userId` | UUID (PK, FK) | Связь с таблицей пользователей (1:1) |
| `bio` | Text | Описание, философия, подход к тренировкам |
| `specialization` | Array<Enum> | Специализация: `MARATHON`, `SPRINT`, `TRAIL`, `RECOVERY`, `GENERAL` |
| `experienceYears` | Integer | Стаж работы/тренировок (лет) |
| `certificates` | JSONB | Список сертификатов (название, дата, организация) |
| `socialLinks` | JSONB | Ссылки: Instagram, Telegram, Strava |
| `rating` | Float | Рейтинг (0.0 - 5.0). *Задел на будущее* |
| `isVerified` | Boolean | Флаг верификации администрацией |
| `createdAt` | Timestamp | Дата создания профиля |

---

### 2.2. Библиотека Тренировок (`Workout`)
Шаблоны тренировок, которые тренер создает один раз и переиспользует в расписании.

**Таблица:** `workouts`

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID (PK) | Уникальный ID тренировки |
| `authorId` | UUID (FK) | ID тренера (User) |
| `clubId` | UUID (FK) | ID клуба (если тренировка принадлежит клубу, опционально) |
| `name` | String | Название ("Интервалы 8x400", "Восстановительная") |
| `description` | Text | Текстовое описание задачи |
| `type` | Enum | `RECOVERY`, `TEMPO`, `INTERVAL`, `FARTLEK`, `LONG_RUN` |
| `difficulty` | Enum | `BEGINNER`, `INTERMEDIATE`, `ADVANCED`, `PRO` |
| `targetMetric` | Enum | `DISTANCE`, `TIME`, `PACE` (Основная цель) |
| `structure` | JSONB | Структурированный план (см. раздел 2.2.1) |
| `isPublic` | Boolean | Видимость для других тренеров (default: false) |

#### 2.2.1. Структура тренировки (JSON)
Пример поля `structure`:
```json
{
  "totalDistanceEst": 8200, // метров (расчетное)
  "totalDurationEst": 3600, // секунд (расчетное)
  "steps": [
    { 
      "order": 1,
      "type": "warmup", 
      "duration": 600, // 10 минут
      "notes": "Легкий бег + суставная разминка" 
    },
    { 
      "order": 2,
      "type": "interval", 
      "repeats": 8, // Количество повторов
      "work": {
        "distance": 400,
        "targetPace": "3:45", // мин/км
        "zone": 4 // Пульсовая зона (опционально)
      },
      "recovery": {
        "time": 90, // 1:30 отдыха
        "type": "active" // активный отдых (трусца)
      }
    },
    { 
      "order": 3,
      "type": "cooldown", 
      "duration": 600, 
      "notes": "Заминка + растяжка" 
    }
  ]
}
```

### 2.3. Интеграция с Событиями (`Event`)
Расширение существующей таблицы событий для связи с библиотекой тренировок.

**Изменения в таблице:** `events`

| Поле | Тип | Изменение |
|---|---|---|
| `workoutId` | UUID (FK) | Ссылка на шаблон тренировки (Nullable) |
| `trainerId` | UUID (FK) | Ответственный тренер на конкретную дату (Nullable) |
| `actualDistance` | Integer | Фактическая дистанция (если отличается от шаблона) |

## 3. Пользовательские сценарии (User Stories)

### 3.1. Тренер: Создание профиля
1. Пользователь с ролью `TRAINER` в любом клубе получает доступ к разделу "Кабинет тренера".
2. Заполняет `bio`, стаж, специализацию.
3. Профиль становится доступен для просмотра участникам клуба (через список участников или детали события).

### 3.2. Тренер: Создание шаблона (Конструктор)
1. Тренер заходит в "Мои тренировки" -> "Создать".
2. Выбирает тип: "Интервалы".
3. Добавляет шаги: Разминка (10 мин) -> Работа (1 км) -> Отдых (3 мин) -> Заминка.
4. Сохраняет как "Фартлек: Пирамида".

### 3.3. Тренер: Планирование (Расписание)
1. Тренер открывает календарь клуба.
2. Нажимает "Добавить тренировку".
3. Выбирает "Из библиотеки" -> "Фартлек: Пирамида".
4. Указывает дату/время и локацию.
5. Система автоматически заполняет описание и теги события данными из шаблона.

### 3.4. Атлет: Просмотр и Выполнение
1. Участник видит событие в ленте.
2. В карточке события отображается детальная структура (график или список отрезков).
3. Видит профиль тренера, который ведет занятие.
4. После завершения пробежки (`Run`), если она была привязана к событию, видит сравнение "План / Факт" (в будущих версиях).
