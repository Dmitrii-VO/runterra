# Исследование и план реализации системы зон (территорий) — Runterra MVP

## 1. Резюме текущего состояния
*   **Backend:** Территории хранятся как статический массив в `territories.config.ts`. Модель данных содержит только координаты центра (`latitude`, `longitude`).
*   **Mobile:** В `MapScreen.dart` территории отображаются через `CircleMapObject` с фиксированным радиусом 500 метров.
*   **Проблема:** Круги не позволяют создать бесшовное покрытие города. Возникают наложения или пустые зоны, что делает логику "захвата территории" неоднозначной на границах.

## 2. Анализ вариантов сетки

| Критерий | Квадратная сетка (MVP) | Гексагональная сетка (H3) |
| :--- | :--- | :--- |
| **Сложность расчетов** | Низкая (простая арифметика) | Средняя (нужны библиотеки типа H3) |
| **Визуализация** | Строгий, квартальный стиль | Игровой, естественный вид |
| **Проверка попадания** | `x >= min && x <= max` | Алгоритм "точка в полигоне" |
| **Адаптивность** | Легко делить квадраты на под-квадраты | Сложно менять масштаб динамически |

**Решение для MVP:** Использовать **адаптивную квадратную сетку**. Это позволит быстро реализовать логику без внедрения сложных ГИС-библиотек на первом этапе.

## 3. Рекомендации по адаптивности (Deep Think)
Для реализации требований "мелкая сетка в парках / крупная в спальниках" без сложного ГИС-анализа, предлагается использовать **статические зоны плотности**:
1.  Определить 3-4 полигона (Bounding Boxes) для центра Санкт-Петербурга и крупных парков (Крестовский, Сосновка, Парк 300-летия).
2.  Скрипт генерации проверяет центр ячейки:
    *   Внутри "Парковой зоны" -> размер ячейки 400м.
    *   Внутри "Центра" -> размер ячейки 700м.
    *   Остальное -> 1200м.

## 4. Поэтапный план реализации

### Фаза 1: Обновление контрактов (Backend)
*   Добавить необязательное поле `geometry: GeoCoordinates[]` в `TerritoryViewDto`.
*   Это позволит фронтенду рисовать полигоны там, где они есть, и оставлять круги-заглушки там, где данных пока нет.

### Фаза 2: Скрипт генерации (Data)
*   Написать TS-скрипт `scripts/generate-territories.ts`.
*   Скрипт должен "нарезать" город на квадраты, учитывая зоны плотности, и выдать готовый JSON для вставки в `territories.config.ts`.

### Фаза 3: Визуализация (Mobile)
*   В `MapScreen.dart` внедрить `PolygonMapObject`.
*   Добавить логику: `if (territory.geometry != null) { drawPolygon() } else { drawCircle() }`.

## 5. Предложение по немедленной реализации (Quick Win)

Чтобы сразу увидеть результат, предлагается расширить `territories.config.ts` функцией-генератором квадратов и обновить мобильное приложение для их отрисовки.

### Изменения в Backend (`territories.config.ts`):
```typescript
function generateSquareGeometry(lat: number, lon: number, sizeInMeters: number) {
  const offset = (sizeInMeters / 111320) / 2;
  return [
    { latitude: lat - offset, longitude: lon - offset },
    { latitude: lat - offset, longitude: lon + offset },
    { latitude: lat + offset, longitude: lon + offset },
    { latitude: lat + offset, longitude: lon - offset },
  ];
}
```

### Изменения в Mobile (`map_screen.dart`):
Замена `CircleMapObject` на `PolygonMapObject` (используя Yandex MapKit `Polygon`):
```dart
PolygonMapObject(
  mapId: MapObjectId('poly_${territory.id}'),
  polygon: Polygon(
    outerRing: LinearRing(points: geometryPoints),
    innerRings: [],
  ),
  fillColor: _getTerritoryColor(territory.status),
  strokeColor: _getTerritoryStrokeColor(territory.status),
  strokeWidth: 1.5,
)
```

## 6. Риски
1.  **Производительность:** Большое количество полигонов на карте. *Митигация:* Использовать упрощенную геометрию (только 4 точки) и фильтрацию на стороне API по текущему вьюпорту.
2.  **Точность:** На больших расстояниях квадраты будут искажаться. *Митигация:* Для масштабов города (до 50км) искажениями в MVP можно пренебречь.
