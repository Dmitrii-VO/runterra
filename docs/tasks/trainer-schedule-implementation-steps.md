# План реализации: Trainer Schedule & Plans (Step-by-Step)

**Основная спецификация:** [docs/features/trainer-schedule-plans.md](../features/trainer-schedule-plans.md)

Этот документ разбивает реализацию функционала "Расписание и Планы" на конкретные технические задачи.

---

## Этап 1: Фундамент и Очистка (Backend) [DONE]
**Цель:** Привести данные в порядок и внедрить правило "Один клуб".

1.  **[x] Миграция "Cleanup Memberships":** 
    *   Написать SQL-скрипт, который находит пользователей с несколькими `active` записями в `club_members`.
    *   Оставить только самую старую запись (`created_at ASC`), остальные перевести в `inactive`.
2.  **[x] Валидация в API:**
    *   Обновить `POST /api/clubs/:id/join` — добавить проверку: если у юзера уже есть активный клуб, возвращать `400 already_in_another_club`.
    *   Обновить `POST /api/clubs` — та же проверка при создании.
3.  **[x] Тестирование:** Написать тесты на запрет двойного членства.

---

## Этап 2: Инфраструктура базы данных (Backend) [DONE]
**Цель:** Подготовить таблицы для хранения шаблонов и связи с календарем.

1.  **[x] Миграция "Schedule Tables":**
    *   Создать таблицу `weekly_schedule_items` (клубные шаблоны).
    *   Создать таблицу `personal_schedule_items` (личные шаблоны).
    *   Добавить поля в `events`: `template_id`, `generated_for_date`, `is_manually_edited`, `deleted_at`.
    *   Добавить поле в `activities`: `scheduled_item_id`.
2.  **[x] Индексы:** Создать Unique Constraints на `(template_id, generated_for_date)` для защиты от дублей.

---

## Этап 3: Основная логика и API (Backend) [DONE]
**Цель:** Дать тренеру возможность управлять шаблонами через API.

1.  **[x] Schedule API:**
    *   `GET /api/clubs/:id/schedule` — получение шаблона недели.
    *   `POST /api/clubs/:id/schedule` — добавление элемента (Event/Note).
    *   `PATCH /api/clubs/:id/schedule/:itemId` — обновление (с логикой `updateFutureEvents`).
2.  **[x] Roster & Personal API:**
    *   `GET /api/clubs/:id/members` — расширить, чтобы видеть тип плана юзера.
    *   `POST /api/clubs/:id/members/:userId/personal-schedule` — настройка личного плана.
3.  **[x] Calendar API (Runner):**
    *   `GET /api/clubs/:id/calendar?month=YYYY-MM` — агрегированный эндпоинт, отдающий Events + Notes.

---

## Этап 4: Автоматизация (Cron Job) [DONE]
**Цель:** Реализовать "вечный календарь".

1.  **[x] Скрипт генерации:**
    *   Реализовать фоновую задачу, которая каждую ночь создает записи на `Сегодня + 31 день`.
    *   Обработка `deleted_at`: если запись была удалена, крон её не пересоздает.
2.  **[x] Триггер "On Save":**
    *   Логика массового обновления/создания записей при изменении шаблона тренером.

## Этап 5: Интерфейс Тренера (Mobile)
**Цель:** Удобное управление клубом.

1.  **Экран "Weekly Schedule":**
    *   Список из 7 дней.
    *   Добавление Note (Текст) или Event (из библиотеки).
2.  **Экран "Club Roster":**
    *   Список участников.
    *   Переход к личному расписанию участника.

---

## Этап 6: Интерфейс Бегуна (Mobile)
**Цель:** Удобный просмотр плана.

1.  **Calendar Strip:** 
    *   Горизонтальная лента дней на экране `Events`.
    *   Логика переключения дат и загрузки данных на месяц.
2.  **Mixed Content Display:**
    *   Отображение карточек Событий и карточек Заметок (Отдых/Задание).
    *   Выделение личных заданий (цветом/иконкой).

---

## Этап 7: Связь План -> Факт (Full Stack)
**Цель:** Привязка реальных пробежек к плану.

1.  **Mobile (Run Tab):**
    *   При нажатии "Start Run" предлагать выбрать сегодняшнее задание.
2.  **Backend (Activity Link):**
    *   Сохранение связи `activity.scheduled_item_id`.
3.  **UI Feedback:**
    *   В календаре карточка задания помечается "Выполнено" (зеленая галочка).

---

## Этап 8: Уведомления (Notifications)
**Цель:** Оперативное информирование об изменениях.

1.  **Push Logic:**
    *   Отправка пуша при отмене тренировки.
    *   Отправка пуша при изменении времени ближайшей тренировки.
    *   Персональный пуш при обновлении личного плана тренером.

---

## Валидация и Приемка
*   Все 8 этапов покрыты тестами.
*   Отсутствие дублей в календаре при работе Крона.
*   Корректная работа часовых поясов.
*   Бегун видит личный план вместо общего (если он есть).
