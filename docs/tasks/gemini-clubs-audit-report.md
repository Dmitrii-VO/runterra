# Аудит и план развития модуля «Клубы» — Runterra

**Модель для анализа:** Gemini 3 Pro
**Дата:** 13 февраля 2026 г.

## 1. Резюме
Текущая реализация модуля «Клубы» представляет собой работающий скелет, который требует доработки логики состояний и синхронизации с модулем территорий. Переход на UUID-идентификаторы создал конфликт со статическими данными, который необходимо устранить для корректной работы карты. Внедрение механики «авто-активации» при достижении 2-х участников позволит оживить игровой процесс без участия модератора.

## 2. Соответствие Product Spec и новым решениям

| Требование | Статус | Комментарий (Решение) |
| :--- | :--- | :--- |
| **Активация клуба** | 🛠 В работе | Авто-перевод `PENDING -> ACTIVE` при наличии **2 членов**. |
| **Захват территорий** | 🛠 В работе | Разрешен только для клубов в статусе `ACTIVE`. |
| **Роль лидера** | ✅ Реализовано | Авто-понижение старого лидера до `trainer` при передаче прав. |
| **Меркатели** | 💡 Дизайн | Пользователи без клуба, помогающие в захвате зон. |
| **ID клубов** | ⚠️ Конфликт | Требуется замена legacy-строк (`club-1`) на UUID или `null`. |
| **Метрики MVP** | 💡 Дизайн | Рейтинг = `(участники * 1) + (территории * 10)`. |

## 3. Технический аудит проблем

### [Критично] Формат ID и Связность (P0)
*   **Проблема:** Бэкенд требует UUID (`isValidUuid`), но конфиг территорий (`territories.config.ts`) ссылается на `club-1`. Это ломает отображение владельцев на карте.
*   **Решение:** Свободные территории в конфиге должны иметь `clubId: null`. Тестовые клубы в БД должны иметь фиксированные UUID, которые прописаны в конфиге территорий.

### [Важно] Логика состояний (P1)
*   **Проверка членства:** Сейчас клуб может быть активным с 1 участником.
*   **Решение:** При создании статус всегда `PENDING`. Как только `ClubMembersRepository.countActiveMembers` достигает 2, статус клуба переключается на `ACTIVE`.

### [Важно] Управление ролями (P1)
*   **Лидерство:** Текущий API `PATCH /role` позволяет иметь несколько лидеров.
*   **Решение:** Обновить `updateRole` на бэкенде: если новому пользователю ставится роль `leader`, найти старого лидера и сменить его роль на `trainer`.

## 4. План реализации (Ближайшие шаги)

### Шаг 1: Исправление данных (Data Integrity)
1.  Очистить `territories.config.ts` от строковых `club-1/club-2`, заменив их на `null` или реальные UUID.
2.  Обновить `005_seed_example_events.sql` для использования валидных UUID.

### Шаг 2: Бизнес-логика (Backend)
1.  **Авто-активация:** В методе `approveMembership` репозитория добавить проверку: если кол-во участников >= 2, обновить `clubs.status = 'active'`.
2.  **Захват территорий:** В логике `checkCaptureConditions` (модуль территории) добавить фильтр: `club.status === 'active'`.
3.  **Передача лидерства:** В роуте `PATCH /members/:userId/role` реализовать атомарную смену ролей (Demote old leader -> Promote new).

### Шаг 3: Метрики MVP (Backend)
1.  Реализовать упрощенный расчет рейтинга в `GET /api/clubs/:id`:
    *   `membersCount` (уже есть).
    *   `territoriesCount` (запрос в `territories` по `clubId`).
    *   `cityRank` = расчетная величина на основе весов (участник=1, территория=10).

### Шаг 4: Меркатели (Logic)
1.  Пользователи без `club_id` могут выполнять пробежки, которые засчитываются в «активность зоны», делая её **оспариваемой (contested)**, что упрощает последующий захват любым клубом.

## 5. Риски
*   **Race Condition:** При одновременном вступлении нескольких участников в `PENDING` клуб. Требуется транзакционность при смене статуса клуба.
*   **UX:** Пользователь может не понять, почему его клуб не может захватить зону. Нужно добавить подсказку в мобильное приложение: "Наберите еще 1 участника, чтобы активировать клуб".
