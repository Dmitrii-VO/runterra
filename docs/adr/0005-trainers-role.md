# ADR-0005: Роль тренера (trainer) в клубах

## Статус

Проектирование (design only)

## Контекст

Роль `trainer` уже существует в БД (миграция 014) и может быть назначена лидером клуба через PATCH /api/clubs/:id/members/:userId/role. Однако **никаких проверок прав тренера в коде нет** — только `leader` имеет привилегии в текущей реализации.

Нужно зафиксировать права тренера для будущей реализации.

## Решение

### Иерархия ролей

| Действие | member | trainer | leader |
|---|---|---|---|
| Читать чат клуба | ✅ | ✅ | ✅ |
| Отправлять сообщения | ✅ | ✅ | ✅ |
| Видеть заявки на вступление | ❌ | ✅ | ✅ |
| Одобрять/отклонять заявки | ❌ | ✅ | ✅ |
| Создавать каналы (подчаты) | ❌ | ✅ | ✅ |
| Создавать события от имени клуба | ❌ | ✅ | ✅ |
| Редактировать клуб (название, описание) | ❌ | ❌ | ✅ |
| Менять роли участников | ❌ | ❌ | ✅ |
| Передать лидерство | ❌ | ❌ | ✅ |
| Распустить клуб | ❌ | ❌ | ✅ |

### Экраны и действия тренера (MVP)

1. **Заявки на вступление** — тренер видит и может одобрить/отклонить (уже реализовано в Z4)
2. **Создание событий** — тренер может создавать тренировки от имени клуба
3. **Каналы** — тренер может создавать подчаты (уже реализовано в Z5)
4. **Управление тренировками** — назначать участников, отмечать результаты (future)

### Приоритет для MVP

- **Высокий**: одобрение заявок (Z4), создание каналов (Z5) — уже реализовано
- **Средний**: создание событий от имени клуба
- **Низкий**: управление тренировками, расписание, статистика группы

### Технические изменения (future)

1. В backend проверках заменить `role === 'leader'` на `['leader', 'trainer'].includes(role)` где нужно
2. На mobile — показывать UI управления заявками для trainer (уже сделано в Z4)
3. Добавить возможность создания событий для тренеров

## Последствия

- Роль тренера становится промежуточной между участником и лидером
- Лидер остаётся единственным, кто может менять структуру клуба (название, роли, роспуск)
- Тренер получает операционные права (заявки, события, каналы)
