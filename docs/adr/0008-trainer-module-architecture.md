# ADR-0008: Архитектура модуля тренера

## Контекст

Роль `trainer` в клубах была номинальной (только запись в `club_members.role`, миграция 014). Спецификация `docs/trainer-implementation-spec.md` описывает превращение тренера в функциональную единицу с профилем, библиотекой тренировок и интеграцией с событиями.

Ключевые архитектурные вопросы:
1. Как проверять право доступа к функциям тренера?
2. Как управлять видимостью тренировок (личные vs клубные)?
3. Кто может назначать тренера на событие?

## Решения

### 1. Хелпер `isTrainerInAnyClub` вместо middleware

**Решение:** Проверка роли тренера реализована как хелпер-функция (`src/api/helpers/trainer-role.ts`), а не как middleware.

**Обоснование:** Не все эндпоинты тренерского модуля требуют проверки роли (например, `GET /api/trainer/profile/:userId` — публичный). Middleware применялся бы ко всем роутам, а хелпер вызывается точечно. Три функции:
- `isTrainerInAnyClub(userId)` — для создания/редактирования профиля и тренировок
- `isTrainerOrLeaderInClub(userId, clubId)` — для создания клубных тренировок
- `isLeaderInClub(userId, clubId)` — для назначения trainerId на событие

### 2. Видимость тренировок: clubId как разделитель

**Решение:** Тренировки с `clubId = NULL` — личные (видны только автору). Тренировки с `clubId` — клубные (видны всем членам клуба).

**Обоснование:** Простая модель без отдельного поля `isPublic`. MVP не требует публичных тренировок вне контекста клуба. При необходимости `isPublic` можно добавить позже (отмечено в спеке как Future).

### 3. Назначение trainerId — только leader

**Решение:** `trainerId` на событие может назначить **только leader** клуба-организатора (спека 4.2). Тренер может назначить только `workoutId`.

**Обоснование:** Назначение человека ответственным за событие — организационное решение. Тренер не должен иметь возможность назначить себя на чужое событие. Leader несёт ответственность за клуб и его мероприятия.

### 4. Защита от удаления привязанной тренировки

**Решение:** `DELETE /api/workouts/:id` возвращает 409 (`conflict`), если тренировка привязана к будущим событиям (`hasUpcomingEvents`).

**Обоснование:** Удаление тренировки с привязанными событиями нарушит целостность данных. Тренер должен сначала отвязать тренировку от событий.
