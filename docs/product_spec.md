# RunMap / Territory Run

## Техническое задание
**Платформа локальных беговых сообществ с геймификацией территорий**

---

## 1. Общее описание проекта

**Рабочее название:** RunMap / Territory Run (placeholder)
**Формат:** мобильное приложение (Android) + веб-админка
**Цель:** собрать беговые клубы, тренеров, забеги и регулярные активности в одном месте и стимулировать регулярный бег через социальную геймификацию.

---

## 2. Проблема

- Беговые активности разрознены по соцсетям, чатам и разным платформам
- Нет единой карты сообществ и событий
- Существующие беговые приложения ориентированы на индивидуальные метрики, а не на локальные коммьюнити
- Низкая мотивация к регулярному участию вне личных целей

---

## 3. Целевая аудитория

### Основная
- Беговые клубы
- Тренеры и организаторы тренировок
- Активные бегуны (социально ориентированные)

### Вторичная
- Любители, ищущие компанию для бега
- **Меркатели** — независимые бегуны, помогающие разным клубам
- Организаторы забегов
- Бренды и партнёры (на следующих этапах)

---

## 4. Ключевая концепция

**Платформа = Карта + Сообщества + Игра**

Основной геймплей — **захват и удержание территорий** через регулярную групповую активность.

---

## 5. Функциональные требования (MVP)

### 5.1 Карта (основа продукта)

**Интерактивная карта города**

**Отображение:**
- территорий (районы / парки / зоны)
- активных клубов
- текущих и предстоящих тренировок

**Фильтры:**
- по дате
- по уровню подготовки
- по клубу / тренеру

### 5.2 Территории

**Описание**
Территория — географическая зона, соответствующая административному району Санкт-Петербурга.

**Логика захвата**
- Территория захватывается клубом
- **Предлагаемые условия захвата:**
  - **5 пробежек** в зоне (с GPS-трекингом, минимум 1 км в пределах зоны)
  - **3 уникальных участника** клуба
  - за период **7 дней**
- При одновременном выполнении условий несколькими клубами — побеждает клуб с большим количеством уникальных участников

**Логика удержания**
- Требуется регулярная активность (минимум **2 пробежки в неделю** с 2+ участниками)
- При отсутствии активности **3 дня** территория становится **оспариваемой** (могут захватить конкуренты с облегченными условиями)
- При отсутствии активности **7 дней** территория становится **нейтральной**

**Отображение**
- Цвет клуба (или нейтральный серый / оспариваемый желтый)
- Название клуба
- Таймер удержания / статус
- История захватов (кратко, последние 3 смены владельца)

### 5.3 Клубы

**Создание клуба:**
- Любой пользователь может создать клуб
- **Требования для создания:**
  - Подтвержденный email/телефон
  - Минимум 3 участника для активации (создатель + 2 приглашенных)
  - Заполненный профиль клуба
- **Модерация:** обязательное одобрение администратором (проверка на дубликаты, валидность)

**Профиль клуба:**
- название
- описание
- город
- зоны активности
- расписание
- основатель/лидер клуба

**Статусы:**
- **pending** — ожидает модерации
- **active** — активный клуб
- **inactive** — нет активности 30+ дней (автоматически)
- **suspended** — заблокирован администратором

**Метрики (MVP):**
- количество участников
- количество удерживаемых территорий
- рейтинг в городе

**Социальные функции:**
- Чат клуба (текстовый, базовый)
- Комментарии к тренировкам

**Управление клубом:**
- Лидер клуба может:
  - Редактировать профиль
  - Назначать модераторов клуба
  - Удалять участников (с подтверждением)
  - Передавать роль лидера
- Модераторы клуба могут:
  - Создавать тренировки
  - Модерировать чат
  - Приглашать участников

**Жизненный цикл "мёртвых" клубов:**
- При отсутствии активности 30 дней — статус меняется на **inactive**
- При отсутствии активности 60 дней — все территории переходят в нейтральный статус
- Клуб можно реактивировать (если лидер создаст хотя бы 1 тренировку)

### 5.4 Тренеры / организаторы

**Профиль:**
- имя
- специализация
- привязка к клубу (опционально)

**Создание тренировок:**
- место (с геометкой на карте)
- дата / время
- уровень сложности
- открытая / закрытая
- описание маршрута (опционально)

**Верификация:**
- Модерация через админку
- Бейдж "Верифицированный тренер"

### 5.5 Тренировки и активности

**Карточка активности:**
- локация
- организатор
- формат (тренировка / совместный бег)
- требования
- список участников
- комментарии

**Возможность:**
- присоединиться (предварительная запись)
- отметить участие (check-in с GPS-подтверждением)
- комментировать после активности

**GPS-трекинг для check-in:**
- Радиус подтверждения: 200-500 метров от точки старта
- Минимальное расстояние для засчета: 1 км в пределах территории
- Базовая защита от читинга: скорость 3-20 км/ч считается валидной

**Управление тренировками:**
- Организатор может отменить тренировку (минимум за 2 часа до старта)
- Если на тренировку никто не записался — автоматическая отмена за 1 час до старта
- Если записались, но никто не пришёл (0 check-in) — тренировка не засчитывается для захвата территории
- **"No-show" политика:** если пользователь 3 раза подряд записался и не пришёл — ограничение на запись на 7 дней

### 5.6 Пользователь (бегун)

**Регистрация:**
- Email / номер телефона
- Имя и фото (опционально)
- Город

**Профиль:**
- имя
- уровень (опционально: новичок / любитель / опытный)
- принадлежность к клубу (не обязательна при регистрации)
- статус "Меркатель" (может помогать нескольким клубам)

**Минимальная статистика:**
- количество участий в тренировках
- вклад в захват территорий (баллы)
- территории, в захвате которых участвовал

**Первый экран после входа:**
- Профиль пользователя (с быстрым переходом на карту)

**GPS-трекинг:**
✅ **Базовый GPS-трекинг для подтверждения участия в пробежках**
- Отслеживание только факта присутствия в зоне и расстояния
- БЕЗ детального анализа пульса, темпа и продвинутой аналитики в MVP

## 6. Геймификация (MVP-уровень)

### Основные механики
- **Захват территорий** (основной геймплей)
- **Командное соревнование клубов** (рейтинг по городу)
- **Вклад каждого участника** в удержание зоны (баллы активности)
- Отсутствие индивидуального «гринда»

### Роль "Меркатели"
- Независимые бегуны без привязки к одному клубу
- Могут участвовать в тренировках разных клубов
- Их пробежки засчитываются клубу-организатору активности
- **Мотивация:** разнообразие маршрутов, знакомства, гибкость
- **Ограничение:** за одну неделю меркатель может помочь не более 3 разным клубам (против эксплуатации)

### Система баллов (предложение)
- За участие в тренировке: 10 баллов
- За вклад в захват новой территории: 50 баллов
- За удержание территории: 5 баллов/неделю
- Баллы суммируются для клуба и для личного профиля

## 10. Стратегия запуска

### Город-пилот: Санкт-Петербург

**Почему СПб:**
- Развитое беговое комьюнити (множество клубов)
- Компактный центр + парки (Летний сад, ЦПКиО, Новая Голландия)
- Набережные — идеальные беговые зоны
- Средний размер города (не слишком большой для MVP)

**План запуска:**
- **Фаза 1 (подготовка):** Партнёрство с 10-15 клубов-партнёров
- **Фаза 2 (пилот):** Запуск в центральных районах + основные парки
- **Фаза 3 (расширение):** Распространение на спальные районы
- **Длительность пилота:** 1-2 месяца
- **Онбординг:** через тренеров и лидеров клубов

**Приоритетные зоны для старта:**
- Центральный район (Марсово поле, Летний сад)
- Крестовский остров
- Елагин остров
- Набережные Невы и каналов

## 15. Архитектурные решения

### Обработка GPS-треков
1. **Клиент (Flutter)** собирает GPS-точки во время пробежки
2. **Отправка на сервер** при завершении активности (массив координат)
3. **Валидация на бэкенде:**
   - Проверка скорости (3-20 км/ч)
   - Проверка расстояния внутри территории (PostGIS ST_Within)
   - Расчет общей дистанции (PostGIS ST_Length)
4. **Сохранение:** упрощенный трек (Douglas-Peucker algorithm) для экономии места

### Система территорий
- **Генерация сетки:** Скрипт на Python (GeoPandas) генерирует гексагональную или квадратную сетку
- **Хранение в БД:**
  ```sql
  CREATE TABLE territories (
    id UUID PRIMARY KEY,
    name VARCHAR(255),
    geometry GEOMETRY(Polygon, 4326),
    club_id UUID REFERENCES clubs(id),
    status VARCHAR(50), -- neutral, owned, contested
    last_activity_at TIMESTAMP,
    ...
  );
  CREATE INDEX idx_territories_geometry ON territories USING GIST (geometry);
  ```
- **Проверка захвата:** Cron job каждые 1 час проверяет условия (COUNT runs + unique participants)

### Real-time обновления
- **WebSocket соединение** для:
  - Обновление статуса территорий на карте
  - Новые сообщения в чате клуба
  - Уведомления о захвате

### Масштабирование
- **Партиционирование БД** по городам (когда будет >1 города)
- **Кеширование** часто запрашиваемых данных (список территорий, рейтинг клубов)
- **CDN** для статики (аватары, логотипы)

## 16. Пользовательские сценарии (User Flows)

### 16.1 Онбординг нового пользователя

**Шаг 1: Регистрация**
1. Пользователь открывает приложение
2. Экран приветствия с объяснением концепции (1 слайд: "Захвати город вместе с беговым клубом")
3. Регистрация через email/телефон (OTP)
4. Ввод имени, выбор города
5. Согласие на использование геолокации

**Шаг 2: Первый вход**
1. Открывается экран профиля с подсказками
2. **Если в городе есть клубы:**
   - Показывается список ближайших клубов
   - Предложение присоединиться или создать свой
3. **Если клубов нет:**
   - Предложение создать первый клуб
   - Или остаться меркателем и ждать появления клубов

**Шаг 3: Первое взаимодействие**
- Кнопка "Открыть карту" → показывается интерактивная карта с территориями
- Подсказка: "Синие зоны — территории клубов. Серые — нейтральные. Твой клуб может захватить их!"

### 16.2 Создание первого клуба

1. Пользователь нажимает "Создать клуб"
2. Заполняет профиль клуба (название, описание, зоны активности)
3. Приглашает минимум 2 участников (через ссылку/код приглашения)
4. После 3 участников — клуб отправляется на модерацию
5. Администратор одобряет клуб
6. Клуб появляется на карте, лидер может создать первую тренировку

### 16.3 Участие в первой тренировке

1. Пользователь видит тренировку на карте или в профиле клуба
2. Нажимает "Присоединиться" → предварительная запись
3. Получает push-уведомление за 1 час до старта
4. Приходит на место, нажимает "Начать пробежку"
5. GPS-трекинг запускается в фоне
6. После пробежки нажимает "Завершить" → данные отправляются на сервер
7. Видит подтверждение: "Засчитано! +10 баллов. Вклад в захват территории 'Марсово поле'"

### 16.4 Захват первой территории клубом

1. Клуб проводит 5 тренировок в одной зоне с 3+ уникальными участниками за 7 дней
2. Система автоматически проверяет условия (cron job)
3. Территория меняет цвет на цвет клуба
4. Все участники клуба получают push: "🎉 Территория 'Марсово поле' захвачена! +50 баллов"
5. Лидер клуба видит обновленную статистику

### 16.5 Поиск и присоединение к клубу

1. Пользователь открывает раздел "Клубы"
2. **Фильтры:**
   - По локации (ближайшие)
   - По уровню подготовки
   - По количеству участников
3. Выбирает клуб → видит профиль с расписанием
4. Нажимает "Присоединиться" → запрос отправляется лидеру клуба
5. Лидер одобряет → пользователь добавлен в клуб и чат

## 17. Анти-читинг и модерация

### 17.1 Защита от GPS-подделки

**Проблема:** Пользователи могут использовать GPS-спуферы для фейковых пробежек.

**Решение (MVP):**
1. **Валидация скорости:** 3-20 км/ч (выше 20 км/ч — подозрительно)
2. **Анализ паттернов движения:**
   - Резкие телепортации (прыжки >500м за 1 сек) — отклонение
   - Идеально ровная скорость (признак симулятора) — флаг для ревью
3. **Проверка сенсоров:** Android API позволяет детектировать mock locations (требует разрешение)
4. **Ручная модерация:** Подозрительные треки отправляются на ревью администратору

**Система флагов:**
- 1 подозрительная активность — предупреждение
- 2 подозрительные активности — временная блокировка check-in на 7 дней
- 3 подозрительные активности — постоянный бан

### 17.2 Защита от ботов и мультиаккаунтов

**Меры:**
- Email/телефон верификация обязательна
- Ограничение на количество аккаунтов с одного устройства (device fingerprinting)
- Капча при регистрации (опционально)
- Минимальный "прогрев" аккаунта: нельзя создать клуб сразу после регистрации (нужно участие в 1 тренировке)

### 17.3 Система репортов

**Пользователи могут пожаловаться на:**
- Читинг (фейковые пробежки)
- Неадекватное поведение в чате
- Спам
- Некорректное название клуба

**Обработка репортов:**
- Автоматическое скрытие контента после 3 жалоб (до ручной модерации)
- Модератор рассматривает в течение 24 часов
- Возможные действия: предупреждение, временный бан, перманентный бан

### 17.4 Модерация контента

**Что модерируется:**
- Названия и описания клубов (перед одобрением)
- Комментарии в чате (по жалобам)
- Названия тренировок (автоматический фильтр мата)

**Инструменты:**
- Автоматический фильтр матов и запрещенных слов
- Ручная модерация через админку
- История изменений для аудита

## 18. Технические ограничения и лимиты (MVP)

### 18.1 API Rate Limiting

**Цель:** Защита от DDoS и злоупотреблений

**Лимиты:**
- **Общие запросы:** 100 запросов/минуту на IP
- **Регистрация:** 5 попыток/час на IP
- **Создание клуба:** 1 клуб/день на пользователя
- **Создание тренировки:** 10 тренировок/день на организатора
- **Check-in:** 1 check-in/час на пользователя (нельзя чекиниться на несколько тренировок одновременно)
- **Сообщения в чате:** 30 сообщений/минуту на пользователя

### 18.2 Ограничения на контент

**Клубы:**
- Максимум 500 участников (для бесплатного тарифа)
- Максимум 10 модераторов
- Название: 3-50 символов
- Описание: до 500 символов

**Тренировки:**
- Название: 5-100 символов
- Описание маршрута: до 300 символов
- Максимум 100 участников на одну тренировку (MVP)

**Чат:**
- Максимум 500 символов на сообщение
- Нельзя отправлять более 5 одинаковых сообщений подряд (защита от спама)

**GPS-треки:**
- Максимум 1000 GPS-точек на одну пробежку
- Автоматическое упрощение трека (Douglas-Peucker) до 100 точек для хранения
- Срок хранения детальных треков: 30 дней (потом остается только упрощенный трек)

### 18.3 Размер файлов

**Аватары пользователей / логотипы клубов:**
- Максимум 5 МБ
- Форматы: JPG, PNG, WebP
- Автоматическое сжатие до 500 КБ на сервере

### 18.4 Производительность

**Целевые метрики:**
- Время загрузки карты: < 2 секунд
- Время ответа API: < 500 мс (95 перцентиль)
- Время check-in: < 1 секунда

## 19. Политика хранения данных

### 19.1 GPS-треки

**Полные треки (с деталями):**
- Хранение: 30 дней после пробежки
- Назначение: Валидация, разрешение споров
- После 30 дней → удаление, остается только упрощенная версия

**Упрощенные треки:**
- Хранение: 1 год
- Назначение: История активности пользователя, статистика

**Агрегированные данные:**
- Хранение: бессрочно
- Пример: количество пробежек, территории, захваченные клубом

### 19.2 Сообщения в чате

**История чата:**
- Хранение: бессрочно (пока клуб активен)
- При удалении клуба: архивирование на 90 дней, затем удаление

**Удаленные сообщения:**
- Мягкое удаление (soft delete) с флагом `deleted_at`
- Физическое удаление через 90 дней

### 19.3 Неактивные аккаунты

**Критерии неактивности:**
- Нет входа в приложение 365 дней

**Действия:**
- Email-уведомление за 30 дней до удаления
- Удаление аккаунта и всех персональных данных
- Анонимизация вклада в статистику (пробежки остаются, но без привязки к пользователю)

### 19.4 Резервное копирование

**База данных:**
- **Полный backup:** ежедневно (retention: 30 дней)
- **Инкрементальный backup:** каждые 6 часов (retention: 7 дней)
- **Хранение:** в отдельном регионе (географически разнесено)

**Критичные данные:**
- Конфигурация территорий → отдельный backup перед каждым изменением
- Профили клубов → ежедневный backup
