# Brainstorm: вкладка Профиль MVP (2026-02-13)

Результаты brainstorm по задаче «как довести вкладку Профиль до MVP: что лишнее, что добавить».

## Контекст

### Product spec (раздел 5.6, 7)

**Профиль пользователя:**
- имя
- уровень (опционально: новичок / любитель / опытный)
- принадлежность к клубу (не обязательна при регистрации)
- статус «Меркатель» (может помогать нескольким клубам)

**Минимальная статистика:**
- количество участий в тренировках
- вклад в захват территорий (баллы)
- территории, в захвате которых участвовал

**Первый экран после входа:** Профиль пользователя (с быстрым переходом на карту)

**Приватность (раздел 7):**
- Возможность скрыть профиль от публичного поиска
- Опциональная видимость статистики для других пользователей

**Уведомления (MVP):** In-app — новые комментарии в чате клуба, приглашения в клуб

## Что может быть лишним для MVP

| Элемент | Аргумент «убрать» | Аргумент «оставить» |
|---------|-------------------|---------------------|
| **PersonalInfo (birthDate, country, gender)** | Spec: только имя, уровень, клуб, меркатель. Лишние поля — дата рождения, страна, пол — не в MVP | Редактирование профиля уже есть; поля пригодятся для расширения. Свёрнуто по умолчанию — не мешает |
| **Activity (next/last)** | Spec не требует «ближайшая/последняя активность» | Контекст «что дальше» и «что было» полезен; связывает профиль с событиями и пробежками |
| **Уведомления (пустой список)** | Секция скрыта при `notifications.isEmpty` — ок | — |

**Рекомендация:** Ничего не убирать. PersonalInfo свёрнут, Activity даёт контекст. Сокращение не даёт выигрыша по scope.

## Что добавить или доработать

### 1. Уровень (новичок / любитель / опытный)

| Вариант | Описание | Сложность |
|---------|----------|-----------|
| **A. Добавить в MVP** | Поле в профиле + редактирование | Низкая (backend: поле в users; mobile: EditProfileScreen) |
| **B. Отложить** | Spec: «опционально» | — |

**Рекомендация:** **B** — отложить. Spec помечает уровень как опциональный; для MVP достаточно имени и клуба.

---

### 2. Видимость профиля (приватность)

| Текущее состояние | Действие |
|-------------------|----------|
| Switch в настройках есть, но `onChanged` не реализован | Backend: поле `profile_visible` в users; PATCH /me/profile; Mobile: вызов API при переключении |

**Рекомендация:** Реализовать для MVP — spec явно требует «возможность скрыть профиль от публичного поиска».

---

### 3. Удаление аккаунта

| Текущее состояние | Действие |
|-------------------|----------|
| Кнопка есть, `onDeleteAccount` — заглушка | Backend: DELETE /api/users/me уже есть | Mobile: вызов API + подтверждение + signOut |

**Рекомендация:** Подключить к существующему DELETE /api/users/me. Требование GDPR/права пользователя.

---

### 4. Видимость статистики для других

| Spec | Реализация |
|------|-------------|
| «Опциональная видимость статистики для других пользователей» | Нет в UI |

**Рекомендация:** Отложить. Нужен публичный просмотр профиля (сейчас его нет). Можно добавить после введения «профиль для других».

---

### 5. Геолокация — открытие настроек

| Текущее состояние | Действие |
|-------------------|----------|
| Показ статуса разрешения, но нет перехода в настройки | `openAppSettings()` при тапе — стандартный паттерн |

**Рекомендация:** Добавить `onTap` с `openAppSettings()` для ListTile геолокации.

## Итоговая рекомендация

**MVP профиля = текущий scope + три доработки.**

1. **Добавить:** видимость профиля (backend `profile_visible` + PATCH + mobile Switch).
2. **Добавить:** удаление аккаунта (вызов DELETE /me из Settings).
3. **Добавить:** открытие настроек по тапу на геолокацию.

**Не убирать:** PersonalInfo, Activity — не мешают, дают контекст.

**Отложить:** уровень, видимость статистики.

## Замечания по коду

- `ProfileSettingsSection`: Switch `profileVisible` — `onChanged` пустой; нужна реализация.
- `profileVisible: true` в ProfileScreen захардкожен; нужно брать из API.
- `onDeleteAccount` передаётся как пустой callback; нужен вызов `UsersService.deleteAccount()` (или аналог) + диалог подтверждения.
